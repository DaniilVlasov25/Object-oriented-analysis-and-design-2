# Лабораторная работа № 2
Тема: Паттерн проектирования «Мост» (Bridge)

# Предметная область
Разрабатывается система конфигурации оружия для RPG-игры. Игрок создаёт арсенал персонажа, выбирая тип оружия (меч, лук, посох) и тип урона (физический, магический, огненный).

# Описание проблемы
Проблема заключается в необходимости создания множества комбинаций оружия и типов урона. 
Если реализовывать каждую комбинацию как отдельный класс, возникают следующие проблемы:
- Комбинаторный взрыв классов — для 3 типов оружия и 3 типов урона требуется 9 классов.
- Жёсткая связанность — тип урона «зашит» в класс оружия, невозможно изменить урон на лету.
- Сложность расширения — добавление нового типа оружия требует создания классов для каждого существующего типа урона.

# Решение
Для решения проблемы в проекте реализован структурный паттерн проектирования Bridge (Мост). Суть паттерна заключается в следующем:
- разделение класса на две независимые иерархии — абстракцию (Weapon) и реализацию (Damage).
- связь через композицию — абстракция содержит ссылку на объект реализации.

В проекте созданы следующие классы:
- Damage (Implementor) — интерфейс реализации, объявляющий методы calculateDamage() и getDamageType();
- PhysicalDamage, MagicDamage, FireDamage (ConcreteImplementor) — конкретные реализации типов урона с различными формулами расчёта;
- Weapon (Abstraction) — абстрактный класс оружия, содержащий ссылку на Damage и делегирующий расчёт урона;
- Sword, Bow, Staff (RefinedAbstraction) — конкретные типы оружия с уникальными характеристиками (sharpness, range, manaCost);
- Player (Client) — класс игрока, управляющий инвентарём оружия.

# Вывод
Преимущества:
- Устранение комбинаторного взрыва — вместо 9 классов имеем 6 классов
- Независимое расширение — добавление нового типа урона требует создания 1 класса вместо 3;

Недостатки:
- Усложнение кода — введение дополнительных интерфейсов и классов;
- Избыточность для простых случаев — если типов урона 1-2, паттерн может быть избыточен;
