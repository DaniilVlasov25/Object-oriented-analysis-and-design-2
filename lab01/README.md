# Лабораторная работа № 1
Тема: Паттерн проектирования «Строитель» (Builder)

# Предметная область
Система кастомизации гоночных автомобилей. Разрабатывается информационная система конфигурации транспортных средств в рамках гоночного симулятора. Пользователь (игрок) создаёт автомобиль для участия в заездах, выбирая базовую модель и оснащая её компонентами.

# Описание проблемы
Проблема предметной области заключается в необходимости создания сложных объектов автомобилей, состоящих из множества взаимосвязанных компонентов (двигатель, колёса, броня, скорость), где многие параметры являются опциональными.
Если создавать объекты Car напрямую через конструктор, возникают следующие проблемы:
- Телескопический конструктор — конструктор требует множество параметров (5+), большинство из которых могут не использоваться;
- Сложность расширения — добавление нового параметра требует изменения всех вызовов конструктора;
- Нарушение инкапсуляции — клиентский код должен знать внутреннюю структуру класса Car для его создания;
- Передача null — для опциональных параметров приходится передавать null, что ухудшает читаемость кода.
Таким образом, требуется механизм, который:
- позволяет создавать сложные объекты пошагово;
- обеспечивает возможность создания разных представлений объекта одним и тем же кодом сборки.
- скрывает детали создания объекта от клиентского кода;

# Решение
Для решения проблемы в проекте реализован порождающий паттерн проектирования Builder (Строитель).
Суть паттерна заключается в следующем:
- создание сложного объекта вынесено за пределы его собственного класса;
- объект строится пошагово через вызов методов строителя;
- один и тот же код строительства может производить разные представления объектов.

В проекте созданы следующие классы:
- Car (Product) — создаваемый сложный объект с приватным конструктором;
- CarBuilder (Builder) — абстрактный класс, объявляющий шаги конструирования;
- SportsCarBuilder, OffroadCarBuilder (ConcreteBuilder) — конкретные реализации строителей для разных типов автомобилей;
- Garage (Director) — управляет порядком вызова строительных шагов;
- Player (Client) — инициирует процесс строительства.

Таким образом:
- клиентский код не зависит от внутренней структуры класса Car;
- опциональные параметры реализованы через пропуск вызова методов Set...();
- добавление нового типа автомобиля требует только создания нового класса-строителя.

# Вывод
Преимущества:
- Инкапсуляция — логика создания объекта скрыта в классах-строителях;
- Читаемость — пошаговая сборка понятнее телескопического конструктора;
- Расширяемость — новый тип авто = новый класс Builder, то есть не нужно изменять конструктор, достаточно добавить новый класс.
Недостатки:
- Усложнение кода — с паттерном колучество классов увеличилось
- Избыточность для простых объектов — для объектов с 2-3 параметрами паттерн не оправдан.
